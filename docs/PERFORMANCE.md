# PERFORMANCE.md — KrishiOx

> Documentation-only artifact recording a performance review against seven areas: **JavaScript
> loading, CSS loading, Image optimisation, Caching, PWA caching, Critical rendering path,
> Lighthouse recommendations.** Scope was safe improvements only — no UI redesign, no build
> step introduced, no behavior change. Every fix below is either a loading-priority hint (zero
> functional risk) or a background-fetch reordering (same end state, better sequencing).

## How to read this report

- 🔧 **Fixed** — a real issue was found and corrected.
- ⚠️ **Flagged, not fixed** — a real opportunity exists, but the fix requires a build step or
  carries real risk without tooling to verify it — outside "safe improvements" scope.
- ✅ **Reviewed, compliant** — checked, already doing the right thing.

## 1. JavaScript loading

🔧 **Fixed** (#1). Finding: this app has no bundler (a deliberate, load-bearing project
decision — see DECISIONS.md D1/D15), so every page's JavaScript is a real dependency graph of
small ES modules, each a separate network request. Tracing the actual graphs:

- The 7 content pages (Home, Services, About, Partners, Contact, Terms, Privacy) each load
  `js/main.js`, which transitively imports 11 more files (`app.js`, `branding.js`,
  `contact.js`, `icons.js`, `navigation.js`, `pwa.js`, `regions.js`, `services.js`,
  `storage.js`, `ui.js`, `whatsapp.js`) — 12 files total.
- `booking.html` additionally loads `js/booking.js`, which pulls in the full booking engine
  (`js/booking/model.js`, `validator.js`, `serializer.js`, `delivery-adapter.js`,
  `whatsapp-delivery-adapter.js`, `engine.js`) plus `utils.js` and `villages.js` — 21 files
  total on that page.

Without any hint, the browser can only discover this graph by fetching `main.js`, **parsing**
it to find out what it imports, fetching those, parsing them to find their imports, and so on —
roughly three sequential round trips before the app shell can render, worse on high-latency
rural connections (this app's explicit target — see ARCHITECTURE.md). `<link
rel="modulepreload">` tells the browser about the whole graph immediately, from the `<head>`,
so it can fetch everything in parallel from the start of page load instead of waiting to
discover it level by level.

Each page's hint list was generated by resolving its actual `import` graph (not guessed), then
verified to match exactly — see "Keeping this accurate" below.

✅ **Reviewed, compliant** (unchanged, already correct): every script tag is `type="module"`,
which defers execution until after HTML parsing completes, same as classic `defer` — there is
no render-blocking JavaScript anywhere in the app. Script tags sit at the end of `<body>`, not
in `<head>`. No inline synchronous scripts block parsing.

⚠️ **Flagged, not fixed**: JavaScript files are not minified (no build step exists to do this
safely — see DECISIONS.md D1). Manually minifying ~30 files by hand would be slow to verify and
easy to silently break; not attempted here. The files are individually small (the largest,
`css/style.css` aside, are a few KB each), so the byte-size cost of non-minification is modest
compared to the round-trip-latency problem #1 actually fixes.

## 2. CSS loading

✅ **Reviewed, compliant, no change made**. A single stylesheet (`css/style.css`) is loaded via
a normal render-blocking `<link rel="stylesheet">` in `<head>` — correct behavior, since this
file *is* the app's critical styling (the whole design system), not an optional enhancement, so
deferring it would cause a flash of unstyled content rather than improve anything. No
third-party or Google-hosted fonts are loaded at all (`--font-main` falls through to system
fonts only — see ARCHITECTURE.md §4 and §10) — this eliminates an entire class of
performance/FOUT-FOIT problems for free; there was nothing to fix. `<link rel="stylesheet">`
already sits early in `<head>`, right after the tags that must come first (charset, viewport).

⚠️ **Flagged, not fixed**: CSS is not minified, for the same no-build-step reason as JavaScript.

## 3. Image optimisation

✅ **N/A / already optimal, no change made**. There are zero `<img>` elements anywhere in the
codebase (confirmed in ACCESSIBILITY.md §7) — all visual content is inline SVG or CSS
gradients, so there is no content-image payload to optimize. The only raster assets are the PWA
icon PNGs (`icons/*.png`), all already small (340 bytes–17.5KB) and not loaded as part of
normal page rendering — favicon/manifest icon requests are opportunistic and non-blocking, and
the larger 512px icon is now specifically deprioritized to the "extra" precache tier (#2) since
it isn't needed for any visible UI. Re-compressing these further, or converting to a modern
format, was considered and not done: the files are already small enough that any saving is
negligible, and `apple-touch-icon.png` in particular must stay PNG for iOS compatibility (see
COMPATIBILITY_REPORT.md), so a mixed-format approach would add risk for no measurable benefit.

## 4. Caching

✅ **Reviewed, largely compliant**; ⚠️ **one platform limitation flagged**. GitHub Pages (the
hosting platform — see ARCHITECTURE.md) serves every static file with a fixed `Cache-Control:
max-age=600` and does not support custom HTTP headers or content-hashed/immutable filenames
without a build step — both are out of this project's control from the code alone. The existing
service worker already compensates for this deliberately: navigation requests bypass the HTTP
cache entirely (`fetch(req, { cache: "no-store" })`) so a fresh deploy is seen immediately
rather than waiting out the 10-minute window, and `updateViaCache: "none"` forces the same for
the service worker script itself (see ARCHITECTURE.md §7, DECISIONS.md D5). This was reviewed
and left as-is — it's already the correct mitigation given the hosting constraint.

## 5. PWA caching

🔧 **Fixed** (#2, #3). Finding: the service worker precaches the entire app shell on install —
correct for offline support — but previously did so as one undifferentiated burst
(`Promise.allSettled` over all ~34 files at once), with no distinction between high-value pages
(Booking, Services — the two most-linked destinations in the app, per COMPONENTS.md's bottom-nav
inventory) and low-traffic informational pages (About, Terms, Privacy). This precache already
ran safely in the background — `js/pwa.js` registers the service worker only after the
`window.load` event, specifically so it never competes with the current page's own critical
resources (see ARCHITECTURE.md §6, DECISIONS.md) — but *within* that background work, everything
raced for bandwidth equally. Restructuring into two sequential tiers (core, then extra) means
that if a farmer taps "बुक करें" a few seconds after their first page loads, `booking.html` and
its full engine are much more likely to already be cached, without having waited on About/Terms/
Privacy to finish first. This required updating `dev/check_cache_version.js`'s regex (it
previously only recognized a single `APP_SHELL` array) so the existing CI safety net that
catches a missed `CACHE_VERSION` bump keeps working against the new two-array structure — see
DECISIONS.md D12 for why that guard exists.

✅ **Reviewed, compliant, unchanged**: per-asset precaching via individual `cache.add()` calls
(not `cache.addAll()`, which is all-or-nothing) so one flaky asset on a bad connection doesn't
sink the whole precache; the navigation race strategy (network vs. 4-second timeout); the
manual tap-to-update handoff. All already correct for this app's stated target — poor/flaky
rural connectivity — and none of it needed to change.

## 6. Critical rendering path

✅ **Reviewed, largely compliant**, 🔧 with the JS-loading fix above also being the main lever
available here. No render-blocking JavaScript (§1). CSS is appropriately render-blocking, not
deferrable without visible harm (§2). No external origins exist anywhere in the app (`script-src
'self'` etc. in the Content-Security-Policy — see ARCHITECTURE.md §10), so there is nothing to
`preconnect` to — zero DNS/TLS negotiation overhead for third parties, which is already optimal
and required no action. The JS-rendered header and bottom-navigation (`#siteHeader`,
`#bottomNav`) are empty in the raw HTML until `js/main.js` runs and fills them in, but both
already have fixed CSS heights (`--header-h`, `--bottomnav-h`) regardless of content — so there
is no layout shift as they populate, only a brief empty-shell moment, which the modulepreload
fix (#1) directly shortens.

## Keeping this accurate

The `modulepreload` hint lists (#1) and the service worker's two precache tiers (#2) both
enumerate specific files by hand — like `sw.js`'s precache list already did before this change,
and like `dev/rebrand.js`'s file list already does (see DECISIONS.md D3/D12) — so both are the
same category of thing this project already accepts: mechanical lists that must be kept in sync
by hand when files are added, removed, or an import changes. If a page's `<script type="module">`
entry point's imports change, re-derive that page's `modulepreload` list from the entry file's
(and its dependencies') `import` statements, and re-run the check `dev/check_cache_version.js`
already performs in CI to confirm `CACHE_VERSION` was bumped alongside any precache-list change.
An incomplete or stale `modulepreload` list is not a functional bug — it's purely a hint, so
being incomplete just means a newly added file doesn't get the round-trip-saving benefit yet,
not that anything breaks.
